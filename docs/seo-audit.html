Quick Wins (Fix in < 30 minutes)

1. Blog & guide posts missing OG images HIGH
Blog detail pages (blog/[slug]) and guide detail pages (panduan-wisata/[slug]) do not pass an ogImage prop to MainLayout. Social shares of these pages will show no image or a fallback.
Fix: Pass ogImage from the content's featuredImage field (or a default) to MainLayout in both page templates.
2. Social media links point to generic URLs MEDIUM
Instagram and YouTube footer links target instagram.com/ and youtube.com/ (homepages) instead of the actual Sungkar Group profiles.
Fix: Update Footer.astro social link hrefs to the real profile URLs.
3. Privacy policy & terms pages are missing HIGH
Footer links for "Kebijakan Privasi" and "Syarat & Ketentuan" both point to / (homepage). No actual privacy or terms pages exist. This is a trust signal gap and may affect legal compliance.
Fix: Create kebijakan-privasi and syarat-ketentuan pages (or route them to real content).
4. Homepage OG title has duplicated text HIGH
OG metadata title resolves to "Sungkar Group - Tour and Travel - Tour and Travel" (the suffix is appended twice). Check SEOMeta.astro title suffix logic.

All Findings by Priority

🔴 High Priority (7 issues)

H1. Hreflang double-prefix bug on locale pages HIGH
On non-Indonesian locale pages (e.g. /en/destinations), alternate links for other non-default locales get double-prefixed: /ar/en/destinations instead of /ar/destinations. This causes incorrect language targeting. The bug is in SEOMeta.astro lines ~80-89 — the canonicalPath still includes the current locale prefix, and the code unconditionally prepends /${l}.
Fix: Strip the current locale prefix from canonicalPath before generating alternate URLs. Use the DEFAULT_LOCALE constant instead of hardcoded "id". Consider using getLocalizedPath() from @/lib/i18n.
H2. Missing x-default hreflang tag HIGH
No <link rel="alternate" hreflang="x-default"> is generated. This tag tells Google which page to show when no language matches the user's preference. Without it, language targeting may be inconsistent in international search results.
Fix: Add <link rel="alternate" href="..." hreflang="x-default" /> pointing to the Indonesian (default locale) version, since that's the most complete content set.
H3. Fake / placeholder business data in schemas HIGH
  • Organization schema uses +62-370-XXX-XXXX — this is a placeholder, not a real phone number
  • sameAs URLs are mock links, not the actual social profiles
Google may flag this as inaccurate data and lose trust in the site's structured data.
Fix: Update schemas.ts with real phone number and actual social profile URLs.
H4. Static coordinates in TouristAttraction schema HIGH
All destination pages use the same lat/lng (-8.7667, 116.3500) in their TouristAttraction schema. This means every destination looks like it's at the same location to Google. The destinations data already has per-destination coordinates (check latitude/longitude fields in the data files), but the schema generator doesn't use them.
Fix: Pass per-destination coordinates to generateTouristAttractionSchema() and update the function to accept dynamic lat/lng.
H5. Hardcoded aggregate rating in Product schema HIGH
All tour packages and vehicles use the same aggregateRating: ratingValue: 4.8, reviewCount: 156. This is the same across all packages regardless of actual reviews. Google may penalize inauthentic structured data.
Fix: Remove aggregateRating from packages that don't have actual review data, or implement per-package ratings from a real reviews source.
H6. Blog post layout: empty image in NewsArticle schema HIGH
generateArticleSchema() is called with an empty string "" for the image parameter on blog posts that have no featured image. Google requires a valid image URL for NewsArticle schema.
Fix: Fall back to a default OG image when featuredImage is not set, or omit the image field from the schema.
H7. Orphan pages: privacy & terms don't exist HIGH
Already listed in Quick Wins (#3). Footer links navigate to /, providing a poor user experience and missing a trust signal for conversions.
Fix: Create dedicated pages for Kebijakan Privasi and Syarat & Ketentuan.

🟠 Medium Priority (9 issues)

M1. Breadcrumb home URL on locale detail pages is absolute / instead of locale-prefixed MEDIUM
On locale detail pages (e.g., /en/destinations/[slug]), the breadcrumb home link points to "/" instead of "/en". This sends users to the Indonesian homepage when they expected their locale's homepage.
Fix: Use homeHref variable (already defined in some locale pages) in breadcrumb instead of hardcoded "/".
M2. No Search page (WebSite schema targets nonexistent /search) MEDIUM
The WebSite schema includes a SearchAction pointing to /search?q={search_term_string}, but no search page or functionality exists. This is technically misleading structured data.
Fix: Either implement a search page (even a simple client-side filter), or remove the SearchAction from the WebSite schema.
M3. Duplicate OG image across 6 page types MEDIUM
hero-lombok.webp is reused for: /, /paket-wisata, /ulasan, /kontak, /faq, /sewa-mobil. Distinct OG images per page type improve social share performance.
Fix: Use more specific OG images for each page type (e.g., tour packages image for /paket-wisata, vehicle image for /sewa-mobil).
M4. Accommodation region pages missing explicit OG image MEDIUM
/akomodasi/[slug] and its locale variants don't pass an ogImage prop, falling back to generic defaults.
Fix: Pass a region-appropriate accommodation image as ogImage to MainLayout.
M5. Blog list page (/blog and locale variants) missing description/keyword optimization MEDIUM
The blog lists use hardcoded Indonesian descriptions in the PageHeader. Locale variants in /[locale]/blog/index.astro also hardcode Indonesian text for the eyebrow/title/description. The meta description uses the hardcoded Indonesian string regardless of locale.
Fix: Use locale-aware strings (uiStrings.page.blog etc.) for blog list page content and meta descriptions across all locales.
M6. No public/robots.txt static fallback MEDIUM
robots.txt is generated only at build time by the astro-robots-txt integration. There's no static fallback in public/. If the build integration fails silently, there's no robots.txt at all.
Fix: Add a minimal public/robots.txt as fallback (the build output will override it).
M7. Sitemap lacks i18n configuration MEDIUM
The @astrojs/sitemap integration uses default settings. There's no i18n config for the sitemap to indicate locale variants of the same page. The sitemap also has no custom filter to exclude any unwanted URLs.
Fix: Add sitemap i18n configuration with locales and default locale, and consider a filter to exclude any test/staging URLs.
M8. Non-default locale pages use Indonesian strings in PageHeader MEDIUM
Several locale page variants hardcode Indonesian eyebrow, title, and description text in the PageHeader component instead of using locale-aware strings. For example, /en/destinations/index.astro shows "Destinasi" (Indonesian) as the eyebrow.
Fix: Use uiStrings.page.destinations etc. in all locale page templates instead of hardcoded Indonesian text.
M9. Guide detail pages missing OG image MEDIUM
Both /panduan-wisata/[slug] and /[locale]/travel-guides/[slug] do not pass an ogImage to MainLayout. Social shares of guide content lack images.
Fix: Pass a default guide OG image (or region-specific image) to MainLayout in guide detail page templates.

🟡 Low Priority (6 issues)

L1. Canonical for locale homepage lacks trailing slash LOW
Locale homepage canonical is set to "/en" instead of "/en/". Astro typically normalizes trailing slashes, but inconsistency could cause minor canonical confusion.
Fix: Normalize to ${locale}/ with trailing slash.
L2. Hardcoded "id" string instead of DEFAULT_LOCALE constant LOW
In SEOMeta.astro, the default locale check uses l === "id" instead of the DEFAULT_LOCALE constant from @/lib/i18n. If the default locale ever changes, this will break.
Fix: Import and use DEFAULT_LOCALE from @/lib/i18n.
L3. No image CDN configured LOW
site-config.ts has a TODO comment about adding a CDN origin. Images are served from the same domain. For a travel site with many photos, a CDN would improve load times.
Fix: Implement image CDN (Cloudinary, Cloudflare Images, etc.) and update CDN_ORIGIN.
L4. MS and ZH blog/guide content uses English stubs LOW
Malay and Chinese locale content for blog posts and guides uses English text. While this doesn't break SEO, it provides a poor user experience for native speakers and may increase bounce rate.
Fix: Provide proper Malay and Chinese translations when available.
L5. author varies by locale in blog frontmatter but not set in NewsArticle schema LOW
The author field in blog frontmatter differs by locale (e.g., "Tim Sungkar Group" vs "Sungkar Group Team"), but the NewsArticle schema doesn't include an author property.
Fix: Add author to generateArticleSchema() and pass it from the blog post data.
L6. Only Instagram and YouTube in social links — no Facebook, TikTok, or X LOW
The footer only links Instagram and YouTube. For a travel agency in Indonesia, Facebook and TikTok are commonly used platforms that could drive additional traffic and serve as positive trust signals.
Fix: Add Facebook and TikTok links if the agency has active profiles on those platforms.
